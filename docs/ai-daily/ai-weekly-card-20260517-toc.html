1 autoresearch — AI 自己优化 AI，一块 GPU 就能跑 630 行代码
Karpathy 开源的「AI 自主科研」框架。核心机制：AI Agent 在一个 Git 特性分支上循环「改代码 → 5 分钟训练 → 检查分数 → 更好就提交」，人类只需写好 program.md 任务说明书，然后去睡觉。
# 获取项目 git clone https://github.com/karpathy/autoresearch cd autoresearch # 准备数据（只需跑一次） python prepare.py # 启动 AI 自主研究（一块 GPU 即可） python train.py # AI 会自动在特性分支上迭代，每 5 分钟一轮 # 第二天早上查看实验日志和合并的改进
适合谁
有 GPU 的研究者 / 想让 AI 自动调参的工程师
最快路径
clone → 跑 prepare.py → 编辑 program.md → 启动 train.py
关键约束
AI 只改 train.py，prepare.py 是固定的，确保可复现
∘ @karpathy · github.com/karpathy/autoresearch

4 「复合工程」— 让每单位工作使未来工作更容易 Every.com
痛点：AI 帮你完成当前任务，但下次类似任务还得重新教它。复合工程（Compounding Engineering）的核心：不要让 AI 只是「完成当前任务」，而要让它「让下一个任务更容易」。
# 实践方法：每次让 AI 完成任务后，顺便生成可复用的 Skill 你: 帮我写这份 PRD，然后生成一个 /write-prd 命令， 让以后任何人都能用一句话生成符合我们规范的 PRD。 # AI 会生成两个文件： 1. .claude/skills/write-prd.md # Skill 定义 2. .claude/CLAUDE.md # 引用 Skill # 下次直接用 /claude /write-prd "我的新功能想法"
实际案例
Dan Shipper 的团队用这个方法，2 个工程师 + 15 个 Claude Code 实例就构建了 Quora（AI 邮件助手），已有 2,500 活跃用户
你的行动
从下一个任务开始，让 AI 顺便生成「如何做这件事」的 Skill，存到 GitHub 形成团队知识资产
∘ @danshipper · lennysvault.com
5 LLM Wiki 方法 — 替代 RAG 的新范式，让 LLM 自己建知识库 Karpathy 新概念
Karpathy 提出的个人知识库构建模式：不让 LLM 临时检索，而是让它在摄入知识时就处理、编译、维护一个 Markdown 知识库。比喻：Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。
# 三层架构 Layer 1: Raw Sources # 原始资料，不可变，LLM 只读不写 Layer 2: The Wiki # LLM 生成的 Markdown 文件集合，自动交叉引用 Layer 3: Ingest Workflow # LLM 自动读取 Raw → 更新 Wiki # 为什么比 RAG 好？ RAG: 每次查询都临时检索，没有知识积累，不知道 AI 参考了什么 LLM Wiki: 知识被「编译」一次，持续更新，所有推理过程可审计
获取方式
搜索 "LLM Wiki Karpathy" 找 Gist，或用 Obsidian + Claude Code 自己搭建
上手建议
创建 `raw/` 目录存原始材料，写 `INGEST.md` 告诉 AI 如何提取和交叉引用
关键优势
知识积累、可审计、跨会话记忆——RAG 做不到这三点
∘ @karpathy · gist.github.com/karpathy
6 Claude Dreaming — Agent 从自己的错误中学习，Harvey 完成任务率提升 6 倍 Anthropic 新功能
Anthropic 发布的 Dreaming 功能：AI Agent 在后台回顾过去会话和记忆，提取模式，生成「未来自我可以参考的笔记和 Playbook」。与 Memory 的区别：Memory 保留偏好和上下文，Dreaming 更高层次，跨会话提取模式。
# Dreaming 的核心机制 1. 回顾: Agent 定期回顾过去 N 个会话 2. 提取: 找出重复出现的错误、有效的工作流模式 3. 编写: 将洞察写成纯文本笔记和 Playbook 4. 应用: 未来会话自动参考这些笔记 # 实际效果 Harvey (法律 AI): 任务完成率提升 ~6x Wisedocs (医疗): 文档审查时间减少 50% Netflix: 同时处理数百个构建日志，用多 Agent 编排
不修改模型权重
Dreaming 不是 fine-tuning，它写笔记供未来会话参考，所有记忆都是纯文本，人类可审计
你的行动
在 `CLAUDE.md` 里明确告诉 AI「完成后把经验写入 skills/ 目录」，定期 review AI 生成的笔记
∘ @AnthropicAI, @alexalbert__ · Code with Claude 2026 Keynote

7 「锯齿状智能」— 理解 AI 能力的边界，设计更好的协作流程 Karpathy 概念
Karpathy 在 Sequoia AI Ascent 2026 炉边谈话中提出：AI 的能力分布是「锯齿状的」—— 一边能重构 10 万行代码库，一边会建议你「走路去 50 米外的洗车店」。
# 锯齿状智能的核心解释 可验证的任务 (代码能跑、答案能对): AI 表现超人 不可验证的任务 (常识判断、模糊决策): AI 可能犯低级错误 # 对你的启示：把任务设计成「可验证」的 ❌ 不要说: "帮我决定用哪个方案" # AI 会瞎猜 ✅ 要说: "生成 3 个方案，每个方案列出优缺点，我来决定" # Prompt 模板：弥补锯齿状缺陷 你是一个 [角色]。 你的任务是 [目标]。 约束条件： 1. [硬性约束，可验证] 2. [硬性约束，可验证] 参考我的 Skills: [[skills/xxx.md]] 如果不确定，给出 3 个选项让我选择，不要自己决定。
关键领悟
不要指望 AI「理解」你的业务上下文，除非你能把上下文写进 Prompt/Skill
你的行动
把常用决策模式写成 Skill，让 AI 按可验证的方式工作，减少低级错误
∘ @karpathy · Sequoia AI Ascent 2026
8 Skills 取代 MCP — AI 工作流标准化的未来方向 GitHub 趋势
过去两个月，GitHub Trending 榜首从 `microsoft/mcp` 变成了 `karpathy/karpathy-skills` 和 `mattpocock/skills`。释放的信号：AI 工作流的标准化正在从「工具调用协议」（MCP）转向「领域知识包」（Skills）。
# MCP vs Skills 对比 MCP (Model Context Protocol) - 定义「AI 能调用哪些工具」 - 是能力层的标准化 - 需要写 server 代码 Skills - 定义「AI 在某个领域如何思考、如何行动」 - 是知识层的标准化 - 一个 Markdown 文件即可 # 为什么 Skills 更有优势？ 1. 轻量: 一个 .md 文件，不需要写代码 2. 可版本控制: 和代码一样用 Git 管理 3. 可分享: 扔到 GitHub 上，任何人都能用 4. 可组合: 像乐高积木一样组装和复用 # 当前生态 - Karpathy Skills: 聚焦 AI 编码最佳实践 - Matt Pocock Skills: 聚焦 TypeScript/React 开发 - 社区正在形成 Skills 市场（类似 VS Code 扩展市场）
你的行动 1
去 GitHub 搜 `topic:claude-skills` 看看别人分享了什么
你的行动 2
把你的 CODM 工作流程写成 Skills（比如「赛季海报生成流程」）
你的行动 3
开源你的 Skills，建立个人影响力，让其他人给你提 PR 改进
∘ @karpathy, @mattpocock, @zarazhangrui · GitHub Trending
