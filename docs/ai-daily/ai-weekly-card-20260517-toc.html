3. 「复合工程」— 让每单位工作使未来工作更容易

Dan Shipper提出的方法论，实现知识复利

具体做法：

1. 每次让AI完成工作后，顺便让它把「如何做这件事」写成可复用的Prompt/Skill

2. 把这些Prompt/Skill存到GitHub仓库，变成团队的「知识资产」

3. 下次类似任务，直接调用，不用重新教AI

实际案例：Dan的团队用这个方法，只用了「两个工程师 + 15个Claude Code实例」就构建并发布了Quora（AI邮件助手），已有2,500活跃用户。

*Source: @danshipper*

4. LLM Wiki 方法 — 替代RAG的新范式

Karpathy提出，让LLM自己建知识库

为什么比RAG好：

1. RAG：每次查询都临时检索，没有知识积累，不知道AI参考了什么

2. LLM Wiki：知识被「编译」一次，持续更新，所有推理过程可审计

三层架构：Raw Sources（原始资料） → The Wiki（Markdown知识库） → Ingest Workflow（自动更新）

*Source: @karpathy*

5. Claude Dreaming — Agent从自己的错误中学习

Harvey完成任务率提升6倍，Netflix用多Agent编排

核心特性：Agent在后台回顾过去会话和记忆，提取模式，生成「未来自我可以参考的笔记和Playbook」，不修改模型权重，所有记忆都是纯文本笔记，人类可审计。

实际效果：法律AI公司Harvey任务完成率提升约6倍，医疗文档审查公司Wisedocs文档审查时间减少50%。

*Source: @AnthropicAI, @alexalbert__*

3. 「锯齿状智能」— 理解 AI 能力的边界

Karpathy提出，AI能力分布是「锯齿状的」

核心解释：AI的能力不是均匀的，而是高度依赖「可验证性」和「经济激励」：

1. 可验证的任务（代码能跑、答案能对）：AI表现超人

2. 不可验证的任务（常识判断、模糊决策）：AI可能犯低级错误

Prompt模板：

你是一个 [角色]。
你的任务是 [目标]。
约束条件：
1. [硬性约束，可验证]
2. [硬性约束，可验证]
如果不确定，给出 3 个选项让我选择，不要自己决定。
*Source: @karpathy*

4. Skills 取代 MCP —— AI 工作流标准化的未来方向

GitHub Trending榜首从microsoft/mcp变成karpathy-skills

为什么Skills更有优势：

1. MCP：定义「AI能调用哪些工具」，是能力层的标准化

2. Skills：定义「AI在某个领域如何思考、如何行动」，是知识层的标准化

3. Skills更轻量（一个Markdown文件），更容易分享和版本控制

你的行动：去GitHub搜topic:claude-skills看看别人分享了什么，把你的CODM工作流程写成Skills。

*Source: @karpathy, @mattpocock, @zarazhangrui*
