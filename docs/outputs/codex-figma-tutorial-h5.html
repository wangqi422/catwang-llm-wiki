04 / Good requests

三种高效
请求模板

用“做成什么”“怎么判断好”“要不要动文件”来控制协作节奏。

request templates
创建：帮我做一个登录页，要有移动端适配，风格参考 Apple。
修改：把首页 CTA 改明显一点，不要动导航和底部。
排查：这个按钮点击没反应，找原因并修复，最后跑测试。
解释：用产品经理能听懂的话解释这段逻辑。
公式：目标 + 上下文 + 限制 + 验收方式›

06 / Edit

让 Codex
直接改项目

当目标明确时，不必停在建议。让它读结构、找到位置、修改文件，并保持改动范围可控。

diff preview
- const title = "Untitled"
+ const title = "Codex 使用教程"
- button.disabled = true
+ button.addEventListener("click", nextSlide)
+ verifyResponsiveLayout()
可以直接说：实现它，并在完成后告诉我改了哪些文件。›
07 / Verify

不要只看成稿
还要做验证

让 Codex 跑测试、检查页面、看截图、修布局问题。靠谱的交付来自最后这一轮。

quality checklist
功能可用
移动端不溢出
没有报错
交互符合预期
localhost / preview
请打开预览，检查桌面和手机尺寸，发现问题直接修。›
08 / Review

需要审查时
直接说 Review

如果你想知道代码风险，让 Codex 用审查模式找 bug、回归风险和缺少的测试，而不是泛泛评价。

review board
让问题排在前面，摘要放后面。
帮我 review 这次改动，重点看真实 bug 和测试缺口。›
09 / Handoff

交付时看三件事

改了什么、怎么验证、还有什么风险。好的结尾让你能立刻继续决策。

handoff card
已完成：新增互动 H5，点击推进章节。
验证：检查桌面和移动端布局。
位置：index.html 可直接打开。
建议：上传静态托管后发给别人。
PM
最后一条消息，应该让你知道下一步怎么走。›
